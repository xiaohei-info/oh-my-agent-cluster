# AGENTS.md —— 配合并行开发 skill 的通用工作规范

> 本文件是 `parallel-dev-skills` 的**配套常驻护栏**：一组**项目无关**的工作纪律与
> 代码准则。把它**合并进你项目自己的 `AGENTS.md` / `CLAUDE.md`**（这类文件每次都会
> 被 Agent 加载），让编排者（leader）与被派到卡的 worker/reviewer 在**同一套约束**下工作。
>
> 为什么需要它：编排 skill 的防跑偏模型有一条「**常驻护栏**」——全局核心约束必须放进
> 「每次都会被加载」的地方，而不是埋在长文档里。本文件就是那份护栏的通用底稿。
> **用法见文末「如何采用」。**

---

## 1. 代码品味准则（第一性原则）

1. **好品味**：消除特殊情况优于增加条件分支。从不同角度重写，让边界情况变成正常情况。
2. **Never break userspace**：任何破坏现有行为的改动都是 bug，无论多么「理论正确」。向后兼容神圣不可侵犯。
3. **实用主义**：解决真实存在的问题，不为臆想的威胁过度设计。
4. **简洁执念**：函数短小、只做一件事；超过 3 层缩进就该重构。以最小必要原则实现目标。

> 动手前先问三句：**这是真问题还是臆想？有没有更简单的做法？会破坏什么吗？**

---

## 2. 工作纪律（与防跑偏模型对齐）

跨模块改动、涉及数据结构/API/状态流变化的任务，**必须**遵循：

1. **先规划，后实现**：先把计划收口到「可按步骤执行」——目标与边界、涉及文件、关键约束与**非目标**、分步顺序、验证方式与完成标准。需求有关键歧义先澄清，不要边写边猜。
2. **按 spec / plan 驱动**：实现受当前计划与共享口径约束，不擅自扩大范围。发现 plan 不足，**先更新 plan 再继续**，不要用「实现者的理解」替代正式口径。
3. **测试先行或至少同步**：新行为有新验证，改旧行为有回归验证。至少覆盖主路径、边界、已知风险点。不能只靠目测「应该没问题」。
4. **完成前必须独立验证**：「代码写完」不等于完成。完成必须附**客观证据**（测试/构建/接口调用/页面交互/关键日志/before-after）。无法验证就明确说明原因与风险，不许宣称完成。
5. **不确定先做小规模验证**：陌生 SDK/API、跨系统集成、关键状态流，先做小、快、可丢弃的试验回答「是否可行/风险在哪」，结论回流到正式计划，别让试验代码直接变生产实现。
6. **调试先找根因**：出 bug/联调失败/契约不一致，**先定位根因再修**。禁止不理解就连续试错、同时改多个变量碰运气、用补丁掩盖契约层问题。修完补回归。

---

## 3. 完成的定义（Definition of Done）

一张卡 / 一个改动算「完成」，需同时满足：

- [ ] 落在**计划的范围**内，未顺手做相邻卡的活（守**非目标**）
- [ ] 只**消费共享契约**、未平行重定义；未违反红线
- [ ] 有**新增/回归验证**且**实际跑过**、结果附在交付里
- [ ] 改动分支覆盖达标（默认 ≥ 90%，除非卡上写明理由）
- [ ] PR base 指向**集成分支**，不是主干
- [ ] 「我觉得可以 / 理论上没问题」**不算**完成

## 4. 与并行开发 skill 的配合要点

这套 skill 的并行度来自「契约先行」，配合时守住：

- **接口是地基**：模块间契约（DTO/事件/枚举/错误/状态/跨服务调用）先于实现冻结，且以**代码**存在。下游只 `import`、**禁重定义**——接口漂移要让编译/类型/测试直接挂。
- **对端可以是假的**：契约冻结后，对着 mock/fake 独立开发，不等对端做完。
- **单一事实源**：每条口径只有一个权威出处，其它地方引用它，**禁止平行拷贝**（拷贝必然漂移）。
- **收活靠证据，不靠自述**：reviewer **独立复跑**测试、看真实 diff（`git diff <base>...<head>`）再判决；worker 的 prose 总结只是线索，不据此放行。
- **只读共享态**：reviewer 审阅只读，不动他人工作树（`git show <ref>:<path>` / `git diff`），避免并发踩踏。
- **范围边界显式写**：每张卡列出**非目标**——这是最隐蔽的越界源。

> 角色细则见两个 skill：`parallel-dev-orchestration`（leader 拆图/编排）、
> `parallel-dev-executor`（worker/reviewer 执行与收活协议）。

---

## 5. 如何采用

把本文件的约束接入你的项目，三选一：

1. **合并（推荐）**：将第 1–4 节整合进你项目已有的 `AGENTS.md` / `CLAUDE.md`，与你的
   业务专属约束（架构边界、领域规则等）并列。Agent 每次加载该文件即受约束。
2. **引用**：在你项目的 `AGENTS.md` 顶部加一行指针，如
   `> 通用工作规范见 parallel-dev-skills/AGENTS.md（契约先行 / 完成需证据 / 根因调试）`。
3. **直接放置**：项目还没有 `AGENTS.md` 时，可将本文件拷到项目根作为起点，再叠加业务约束。

> 注意：本文件只放**通用工程纪律**。**项目专属**口径（架构分层、技术选型、领域模型、
> 权限角色、部署等）仍写在你自己的 `AGENTS.md` 里，不要塞进这份通用底稿。
